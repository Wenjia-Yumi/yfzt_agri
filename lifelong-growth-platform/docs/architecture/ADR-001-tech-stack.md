# ADR-001：技术栈选型决策记录

- **状态**：已接受
- **日期**：2024-01-01
- **决策者**：架构团队

## 背景

lifelong-growth-platform 是一个以「知识图谱可视化」为核心交互的终身学习平台。用户的核心体验是：在一张星图中看到知识节点之间的依赖关系，并追踪自己的学习进度。这一需求对技术栈提出了独特要求：

1. 知识节点间的**前置依赖关系**是图结构，而非简单的树状或列表结构
2. **图谱可视化**需要处理数十个节点的力导向布局，并支持缩放、拖拽、高亮
3. AI 推荐功能需要**实时流式响应**，不能有明显等待感
4. 未来需要支持**多专业图谱**，数据结构必须高度可扩展

---

## 决策一：图数据库选择 Neo4j

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| PostgreSQL（JSON列） | 已有依赖，运维成本低 | 图查询性能差，无法高效执行多跳路径查询 |
| **Neo4j**（选用） | 原生图模型，Cypher语言表达力强，多跳路径查询性能优异 | 额外运维成本，团队学习曲线 |
| Amazon Neptune | 托管服务，免运维 | 成本高，冷启动慢，本地开发不友好 |

### 决策理由

「查找从节点 A 到节点 B 的所有可行学习路径」是一个典型的**图遍历问题**（多跳最短路径）。在 PostgreSQL 中需要使用递归 CTE，随着图规模增长性能急剧下降；而 Neo4j 的 Cypher 语言可以用一行表达：

```cypher
MATCH path = shortestPath((start:KnowledgeNode {id: $startId})-[:PREREQUISITE_OF*]->(end:KnowledgeNode {id: $endId}))
RETURN path
```

此外，Neo4j 的社区版足够满足 V1.0 需求，Docker 部署简单，与 NestJS 的 `neo4j-driver` 集成成熟。

---

## 决策二：后端框架选择 NestJS

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| Express.js | 轻量灵活 | 需要大量手动配置，缺少约定，大型项目维护困难 |
| Fastify | 性能极高 | TypeScript 支持不如 NestJS 完善，生态稍弱 |
| **NestJS**（选用） | 强约定，模块化架构，TypeScript 原生，装饰器风格与 Angular 相似，DI 系统完善 | 初始样板代码较多，学习曲线陡峭 |

### 决策理由

平台需要清晰的模块边界（auth/graph/learning/ai/certificate 各自独立演进）。NestJS 的模块系统、装饰器、DTO 验证（class-validator）、Guard/Interceptor 机制与这一需求高度契合，且 TypeORM 和 neo4j-driver 在 NestJS 生态中均有成熟集成方案。

---

## 决策三：前端可视化选择 D3.js

### 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| ECharts | 配置简单，中文文档好 | 定制化能力有限，节点交互不灵活 |
| Cytoscape.js | 专为图/网络设计，API友好 | 视觉风格偏传统，动画效果一般 |
| React Flow | React 原生集成，交互优秀 | 力导向布局不如 D3 强大，付费功能限制多 |
| **D3.js**（选用） | 最强大的数据可视化库，力导向图成熟，完全自定义节点外观与动画 | 学习曲线最陡，需要手动管理 DOM |

### 决策理由

「知识星图」需要实现类似星空的视觉风格（暗色背景、发光节点、动态动画、SVG滤镜），这要求对每个节点的渲染方式有完全的控制权。D3 的 SVG 渲染模式和 filter/glow 特效支持最佳。力导向模拟（d3-force）是实现知识节点自动布局的标准方案。

---

## 决策四：状态管理选择 Zustand

### 决策理由

相比 Redux Toolkit，Zustand 的 API 极简（无需 action/reducer/selector 三层），对 TypeScript 的类型推断友好，bundle 体积小（~1KB）。知识图谱的状态（节点列表、选中节点、过滤器、加载状态）完全可以用 Zustand 的扁平 store 表达，无需引入 Redux 的复杂性。

---

## 后续审查点

- **V1.1**：若用户数超过1万，评估是否需要引入 Redis 缓存 Neo4j 的图查询结果
- **V2.0**：若需要支持实时多人协作，评估 WebSocket（Socket.io）集成方案
- **图谱规模**：若单个图谱节点数超过200，需要评估 D3 力导向图的渲染性能，考虑引入 WebGL（Three.js）渲染层
